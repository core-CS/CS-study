## 왜 2차 캐시는 복사본을 반환할까요?
>2차 캐시가 복사본을 반환하는 이유는 여러 영속성 컨텍스트가 공유하는 캐시이기 때문입니다. 
> 
> 만약 2차 캐시에 저장된 객체 인스턴스를 그대로 반환하면, 한 트랜잭션에서 엔티티를 수정했을 때 다른 트랜잭션이나 영속성 컨텍스트에서도 그 변경이 보일 수 있습니다. 그러면 트랜잭션 격리성이나 영속성 컨텍스트의 독립성이 깨지고, 변경 감지 대상도 모호해집니다. 
> 
>따라서 2차 캐시는 캐시된 객체 자체를 직접 반환하지 않고 복사본을 반환해서, 각 영속성 컨텍스트가 독립적으로 엔티티를 관리하도록 합니다. 이 때문에 1차 캐시는 객체 동일성을 보장하지만, 2차 캐시는 서로 다른 영속성 컨텍스트 간 객체 동일성을 보장하지 않습니다.

### 2차 캐시가 복사본을 반환하는 이유
여러 영속성 컨텍스트가 공유하는 캐시이기 때문!!

#### 만약 2차 캐시에 저장된 객체 인스턴스를 그대로 반환한다면?
한 트랜잭션에서 엔티티를 수정했을 때 다른 트랜잭션이나 영속성 컨텍스트에서도 그 변경이 보일 수 있음!

**-> 트랜잭션이나 영속성 컨텍스트의 독립성이 깨지고, 변경 감지 대상도 모호**

### 왜 락이 아니고 복사본일까?
락은 동시 접근을 제어하는 도구이고, 복사본은 객체 상태의 독립성을 보장하는 도구

1. 락을 걸면 2차 캐시의 성능상 의미 감소
```java
요청 A -> Member 캐시 객체 사용 중
요청 B -> 같은 Member 캐시 객체 접근하려고 대기
요청 C -> 같은 Member 캐시 객체 접근하려고 대기
```

2. 락을 걸어도 영속성 컨텍스트의 독립성 문제 존재

락은 동시에 수정하지 못하게 막을 수 있지만

'이 객체를 어느 영속성 컨텍스트가 관리할 것인가?'는 JPA 구조적 문제는 해결 X

```java
Member member1 = em1.find(Member.class, 1L);
Member member2 = em2.find(Member.class, 1L);

member1.changeName("A");
```
만약 member1 == member2라면

이 변경을 em1의 변경 감지 대상으로 봐야 하는지, em2의 변경 감지 대상으로 봐야 하는지 애매
