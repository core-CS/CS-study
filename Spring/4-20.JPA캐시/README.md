## JPA 1차 캐시, 2차 캐시에 대해 설명해주세요.

>JPA의 1차 캐시는 영속성 컨텍스트 내부 캐시이고, 2차 캐시는 애플리케이션 범위의 공유 캐시입니다.
> 
> 1차 캐시는 같은 트랜잭션 안에서 같은 엔티티를 다시 조회할 때 DB 대신 캐시에서 반환해 객체 동일성을 보장합니다. 하지만 요청이나 트랜잭션이 끝나면 사라지기 때문에 성능 캐시라기보다는 변경 감지, 쓰기 지연, 동일성 보장 같은 JPA 메커니즘을 위한 캐시에 가깝습니다.
> 
> 2차 캐시는 여러 영속성 컨텍스트가 공유할 수 있는 캐시라 조회 성능 개선 목적이 크지만, 복사본을 반환하기 때문에 서로 다른 영속성 컨텍스트 간 객체 동일성은 보장되지 않습니다.

### 1차 캐시
![1차 캐시](https://velog.velcdn.com/images/pgmjun/post/991248c0-c239-4e6d-8cce-f1207cec0448/image.png)
영속성 컨텍스트 내부, 엔티티를 보관하는 저장소 (Key-Value 형태의 구조)

**[특징]**
- 보통 트랜잭션을 시작하고 종료할 때까지 1차 캐시 유효
- OSIV 사용해도 사용자의 요청이 들어올 때부터 끝날때까지만 1차 캐시 유효
- 같은 엔티티가 있으면 객체 동일성 보장

**[1차 캐시 조회]**
1. EntityManager로 DB 조회
2. 영속성 컨텍스트의 1차 캐시 먼저 조회
3. 존재하면 그 값 찾아 리턴
4. 존재하지 않는다면 DB 조회
5. DB 데이터를 1차 캐시에 저장
6. 값 리턴

**[1차 캐시 한계]**

큰 효율성은 X

요청마다 생성되는 영속성 컨텍스트는 요청이 끝나고 트랜잭션이 종료될 때
함께 사라지기 때문에 1차 캐시는 큰 도움이 안되는 경우 존재

1차 캐시의 주요 목적은 **"메커니즘"**

=> 동일성 보장, 변경 감지, 쓰기 지연, 트랜잭션 내 객체 상태 일관성 유지

### 2차 캐시
![2차 캐시](https://velog.velcdn.com/images/bagt/post/1af29594-e1a7-4a8c-95a0-a4d17c37dfae/image.png)

어플리케이션 범위의 캐시 (공유 캐시)

=> 어플리케이션을 종료할 때 까지 캐시 유지

**[특징]**
- 동시성을 극대화하기 위해 키시 한 객체를 직접 반환하지 않고 복사본을 만들어 반환
- 영속성 컨텍스트가 다르면 객체 동일성을 보장하지 않음

**[2차 캐시 동작]**

1. 1차 캐시 찾아보고 없으면
2. 2차 캐시에서 엔티티 찾음
3. 엔티티 존재하면 반환
4. 존재하지 않으면 DB 조회해서 2차 캐시에 결과 저장
5. 결과 반환
