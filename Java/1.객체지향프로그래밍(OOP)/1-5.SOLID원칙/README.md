## SOLID 원칙 (객체 지향 설계의 5원칙)

### 1. SRP (단일 책임 원칙)

한 클래스는 하나의 책임만 가져야 한다.

즉, 하나의 변경 이유에 대해 이 클래스만 변경되어야 한다.

### 2. OCP (개방 - 폐쇄 원칙)

확장에는 열려 있고, 수정에는 닫혀야 한다.

### 3. LSP (리스코프 치환 원칙)

자식 클래스는 언제나 부모 클래스를 대체할 수 있어야 한다.

🚨 위반 예시

`직사각형` 클래스를 상속 받은 `정사각형` 클래스

➡️ 예를 들어 "가로 길이" 를 바꾸는 메서드가 있을 때, 정사각형 클래스는 가로 세로가 모두 바뀌어야 한다.

부모 메서드에서 기대하는 결과값의 범위를 벗어나거나, 예외를 던지면 안 된다.


### 4. ISP (인터페이스 분리 원칙)

사용하지 않는 메서드에 의존하도록 강제하지 않는다.

즉, 인터페이스를 세분화해 분리해야 한다.

🚨 위반 예시

`프린터기` 인터페이스에 `인쇄()`, `팩스()`, `복사()`, 기능이 다 들어 있다면, 만약 인쇄 기능만 필요한 `저가 프린터기`에도 위의 메서드가 모두 구현되어야 한다. (비어있더라도)

우리는 범용 인터페이스 보다 특정 클라이언트를 위한 여러 개의 인터페이스를 만드는 것이 더 유리하다.

### 5. DIP (의존 역전 원칙)

구현체(low-level)이 아닌 추상화(high-level) 에 의존해야 한다.

우리가 결제 서비스를 구현할 때 직접 삼성 페이 클래스를 생성하는 것이 아닌 Payment 라는 인터페이스만 사용해야 한다.

이 Payment 에서 Samsung 또는 Apple 등의 무엇이 들어올지는 Spring 외부에서 주입받아 사용하는 것이다.
