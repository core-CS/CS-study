## 객체 지향 프로그래밍(OOP)와 그 핵심 원칙 (캡슐화, 상속, 다형성, 추상화)


> 💡 **OOP**는 C와 같은 절차 지향 프로그래밍이 아닌 **객체의 관점**에서 프로그래밍을 한다는 방법론입니다. 자바의 경우 구성 단위가 클래스가 됩니다.

> **추상화**는 복잡한 시스템에서 핵심 특징만 추출하는 것,
> **캡슐화**는 private 같은 접근 제어를 통해 외부의 잘못된 접근으로 보호하는 것입니다.
> **상속**은 공통 기능을 공유해 코드의 재사용성을 높이고,
> **다형성**은 하나의 인터페이스나 상위 클래스 타입으로 여러 하위 객체를 다룰 수 있게 해 결합도를 낮추고 확장성을 높여줍니다.



### 1️⃣ OOP란?

객체 지향 프로그래밍.

Object Oriented Programming 의 준말

프로그램을 명령어의 목록으로 보는 시각에서 벗어나, **여러 개의 독립된 단위의 객체**들의 모임으로 파악하는 것

### 2️⃣ OOP의 핵심 원칙 4가지

#### 🤷‍♂️ 추상화 (Abstraction)

불필요한 세부 사항은 숨기고, 사물의 공통적인 특징(속성과 기능)을 추출해 정의하는 과정입니다.

- 추상화 예시

우리는 스마트폰 내부 회로를 알 필요 없이 `call()` 이나 `message()` 같은 인터페이스만 보면 된다.

우리는 `Interface` 나 `Abstract Class` 를 통해 What 만 정의를 하고, How 는 하위 클래스에서 구현을 하게 된다.

#### 💊 캡슐화 (Capsulation)

데이터와 그 데이터를 처리하는 메서드를 하나로 묶고, 외부로부터 데이터를 보호하는 것이다.

우리는 `private` 접근 제어자를 통해 정보를 숨기고, 객체 간 결합도를 낮출 수 있다.

- 캡슐화 예시

우리는 직접 스마트폰의 전압을 조절할 일이 없다. 그저 충전 단자라는 제한 통로로 접근해 충전을 할 수 있을 뿐이다.

#### 🧑‍🧑‍🧒‍🧒 상속 (Inheritance)

이미 정의된 상위 클래스의 특징을 하위 클래스가 물려받아 재사용하는 것이다.

- 상속 예시

새 스마트폰이 출시되었다. 그러나 이를 설계한 사람은 처음부터 끝까지 다 만들었을까? 아닐 것이다.

이전 시리즈의 스마트폰의 설계도를 그대로 물려 받아 대신 `C타입 단자` 이나 `새로운 카메라` 같은 기능만 추가했을 뿐이다.

🚨 단 상속보다는 조합(Composition) 을 더 권장한다. 상속은 클래스 간 결합도가 높아지는 방법이기에 유연성이 떨어질 수 있다.


#### 👽 다형성 (Polymorphism)

하나의 객체가 여러 가지 타입을 가질 수 있는 특징이다.

우리는 이를 Overriding 을 통해 상위 클래스의 메서드를 하위 클래스에서 재정의해 사용하게 된다.

클라이언트는 실제 구현체가 무엇인지 몰라도 동일 인터페이스로 소통할 수 있게 된다.

- 다형성 예시

우리 스마트폰이 갤럭시이든, 아이폰이든, `C타입 충전기` 라는 인터페이스 규격에만 맞으면 어느 핸드폰이든 충전할 수 있다.


``` java
// 1. 추상화: 결제라는 행위를 인터페이스로 정의한다.
public interface PaymentProcess {
    void pay(int amount);
}

// 2. 캡슐화: 결제 금액과 상태를 보호한다.
public class PaymentContext {
    private PaymentProcess process;

    public void setStrategy (PaymentProcess process) {
        this.process = process;
    }
}

// 3. 다형성: 다양한 결제 수단을 구현한다.
@Component
public class KakaoPay implements PaymentProcess {
    @Override
    public void pay (int amount) {
        // 카카오 페이 결제 로직
    }
}

@Component
public class NaverPay implements PaymentProcess {
    @Override
    public void pay (int amount) {
        // 네이버 페이 결제 로직
    }
}

```

### 설계 관점에서의 장단점

[장점]

- 상속/Composition 을 통해 코드를 재사용하기 유리하다.

- 캡슐화를 통해 변경의 영향 범위를 최소화할 수 있다.

- 다형성을 통해 새로운 기능을 추가하기 쉽다.

[단점]

- 그러나 초기 설계에 시간이 걸릴 수 있고

- 절차 지향적 프로그래밍에 비해 함수 호출에 따른 오버헤드가 발생한다.

- 또한 객체 단위의 관리가 메모리 사용량을 늘릴 수 있다.
