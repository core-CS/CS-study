## 추상 클래스와 인터페이스의 차이점, 그리고 언제 써야 할지 (w. Java 8 default method)

> 💡 추상 클래스와 인터페이스의 차이는 **상태 공유**와 **다중 구현**의 여부에 있습니다.

> **추상 클래스**는 하위 클래스들이 공통적으로 가져야 할 필드와 기본 로직을 물려줄 때 사용하며, IS-A 관계입니다.

> **반면 인터페이스**는 클래스의 계층과 상관없이 특정 기능을 수행할 수 있는 CAN-DO 또는 이 기능을 가지고 있다는 HAS-A 의 관례로, 한 클래스가 여러 개의 인터페이스를 동시에 구현할 수 있습니다.


이 둘이 설계 의도가 다르다는 점을 중심으로 이해하면 더 쉽습니다.

### 추상 클래스

Abstract Class

`Is - A` 의 관계로 ~이다. 를 의미합니다.

하위 클래스들이 공통된 특징을 공유하고 기능을 확장할 때 사용합니다.

🤷‍♂️ **언제 사용할까요**

부모와 자식 간의 긴말한 연결이 필요하고, 부모가 상태를 관리해야 할 때 사용합니다.

🚘 자동차라는 추상 클래스가 있습니다.

- 모든 자동차는 연료량과 속도라는 상태를 가집니다.

- 테슬라든 람보르기니든 같은 자동차이므로 이 상태를 공유합니다.

- 자동차라면 연료가 있어야 하고, 엑셀을 밟으면 움직여야 한다. 는 설계도를 물려주게 됩니다.



[1. 상속]

- `extends` 로 단일 상속만 가능합니다.

[2. 변수]

- 일반적인 멤버 변수를 가질 수 있습니다.

[3. 생성자]

- 상태 초기화를 목적으로 가질 수 있습니다.

[4. 접근 제어자]

- `public`, `protected`, `private` 가 가능합니다.




### 인터페이스

Interface

`Has - A`, `Can -Do` ~를 가지다. 또는 ~를 할 수 있다를 의미합니다.

어떤 클래스가 특정 기능을 수행할 수 있다는 자격을 부여할 때 사용합니다.

🤷‍♂️ ** 언제 사용할까?**

서로 다른 계층의 클래스들이 공통된 기능을 가져야 할 때 사용합니다.

🤖 자율 주행 자격증 인터페이스

- 이 자격증은 자동차가 가질 수도 있지만, 비행기나 로봇 청소기 등 자율 주행 기능만 있다면 무엇이든 딸 수 있습니다.

- 즉, 이 자격증을 따려면 스스로 길 찾기 기능이 구현되어 있어야 합니다.


[1. 구현]

- `implements` 로 다중 구현이 가능합니다.

[2. 필드]

- `public static final` 상수만 가능합니다.

[3. 생성자]

- 가질 수 없습니다.

[4. 접근 제어자]

- 기본적으로 전부 `public` 입니다.

```java
// 1. 추상 클래스
abstract class Car {
    int fuel = 100;

    void refill() { fuel = 100; }

    // 실제 주행 방식은 자식이 구현
    abstract void drive();
}

// 2. 인터페이스
interface Swimmable {
    void swim();
}

// 3. 구현체
class Tesla extends Car implements Swimmable {

    @Override
    void driver() {
        System.out.println("전기 주행");
    }

    @Override
    public void swim() {
        System.out.println("테슬라 수중 모드 전환");
    }

}

class Boat implements Swimmable {

    // 자동차가 아니니 상속 이유가 없습니다.
    @Override
    public void swim() {
        System.out.println("프로펠터를 돌려 수영합니다.");
    }
}

```


### Default Method ?

인터페이스 내부에서 메서드 앞에 `default` 키워드를 붙이면, **인터페이스임에도 불구하고 직접 구현 코드를 가질 수 있게 됩니다.**

그래서 이를 구현하는 클래스에서 따로 오버라이딩 없이 사용할 수 있습니다.

😒 **왜 생긴걸까?**

우리가 `Collection` 이라는 인터페이스에 새로운 기능이 추가되면 어떨까요?

- 인터페이스에 메서드를 추가하는 순간 이를 사용하고 있던 모든 클래스는 컴파일 에러가 날 것 입니다. (구현을 안 했으니)

- `default` 메서드를 추가함으로서, 기존 클래스를 건드리지 않고도 공통 기능을 추가할 수 있습니다.

😒 **추상 클래스와 다른점?**

언뜻보면 둘이 유사한 방향으로 흘러간 것 처럼 보이는데,

왜냐하면 "공통된 구현 코드를 자식에게 물려준다." 는 추상 클래스만 가지는 특징 이었습니다.

하지만 인터페이스는 로직을 가질 수는 있다만, 인스턴스 변수는 여전히 가질 수 없습니다. 즉, **상태를 저장하거나 변경 불가**합니다.

추상 클래스는 공통 필드를 통해 하위 클래스의 상태를 **직접 관리가 가능**합니다.

```java
interface SmartDevice {
    // default Method
    default void powerOn() [
        System.out.println("기본 전원을 킵니다,");
    ]
}

class SmartWatch implements SmartDevice {
    @Override
    public void process() {
        ...
    }

    // powerOn 은 구현 없어도 알아서 물려받게 됩니다.
}

```
