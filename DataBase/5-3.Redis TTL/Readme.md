<h2>Redis의 TTL은 어떤 식으로 동작하나요?</h2>

>Redis의 TTL(Time To Live)은 키에 만료 시간을 설정해서 자동으로 삭제되게 하는 기능이다. `EXPIRE` 명령어로 초 단위 만료 시간을 설정하고, TTL 명령어로 남은 시간을 확인할 수 있다.
만료된 키를 삭제하는 방식은 두 가지를 혼합해서 사용한다.
>
>첫 번째는 Lazy Expiration(지연 삭제)이다. 클라이언트가 해당 키에 접근할 때 만료 여부를 확인하고, 만료됐다면 그 순간 삭제한다. CPU 부담이 적지만, 접근하지 않는 만료된 키는 계속 메모리를 차지한다는 단점이 있다.
>
>두 번째는 Active Expiration(적극적 삭제)이다. Redis가 주기적으로(기본 100ms마다) 만료된 키들을 샘플링해서 삭제한다. 이를 통해 접근되지 않는 키도 정리된다.
>
>두 방식을 함께 쓰는 이유는 CPU 사용량과 메모리 효율성의 균형을 맞추기 위해서다. 만약 메모리가 부족해지면 maxmemory-policy 설정에 따라 LRU, LFU 같은 정책으로 키를 추가로 제거한다.
>
>TTL은 주로 캐시 만료, 세션 타임아웃, 인증 토큰 유효기간, Rate Limiting 같은 곳에 활용된다.


<h3>만료 키 삭제 방식</h3>
1. Lazy Expiration
   
    동작 방식: 클라이언트가 키에 접근할 때 만료 여부를 확인하고, 만료됐으면 그 순간 삭제한다.

    ```
   [클라이언트] GET session:abc123
       ↓
    [Redis] 만료 시간 확인
    ↓
    만료됨? → 삭제 후 nil 반환
    아직 살아있음? → 값 반환
   ```

    장점: CPU를 거의 사용하지 않음. 필요한 순간에만 삭제
   
    단점: 접근하지 않는 만료된 키는 계속 메모리를 차지 

    만약 만료된 키가 수백만 개 있는데 아무도 접근하지 않으면, 메모리가 낭비된다. 이걸 해결하기 위해 두 번째 방식이 필요하다.


2. Active Expiration

   동작 방식: Redis가 주기적으로 만료된 키를 찾아서 삭제한다.

    ```
   [Redis 내부 타이머 - 100ms마다 실행]
       ↓
    만료 시간 있는 키 중 랜덤 20개 샘플링
    ↓
    만료된 키 삭제
    ↓
    만료 비율 25% 이상? → 반복
    만료 비율 25% 미만? → 종료
   ```

   장점: 접근되지 않는 키도 정리됨

   단점: CPU를 조금 사용함 (하지만 확률적 샘플링이라 부담 적음)
