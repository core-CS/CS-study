# 인덱스의 내부 원리에 대해 설명해주세요.

특정 컬럼 데이터를 미리 정렬해 별도 공간에 **키(key, 컬럼값)-포인터(ROW ID) 쌍**으로 저장하는 자료 구조 입니다.

### Key

인덱스를 생성할 때 지정한 컬럼의 값 입니다.

예를 들어 `이름`인덱스라면 사과, 망고가 됩니다.

### Pointer

해당 키 값이 위치한 실제 데이터의 물리 주소입니다.

- RID (Row ID)

데이터가 저장된 디스크의 물리적 주소(파일 번호, 페이지 번호, 슬롯 번호) 입니다. 이는 비클러스터링 인덱스에서 사용합니다.

- PK

앞에서 PK 이 자체가 인덱스라고 했던 말 기억하시나요?

MySQL의 InnoDB처럼 클러스터형 인덱스를 사용하는 경우, 보조 인덱스의 포인터는 실제 주소가 아닌 PK 를 가리키기도 합니다.

### 동작 순서

1. 인덱스 리프 노드에 정렬된 Key 중 해당 Index 를 찾습니다.
2. 이 Index옆에는 쌍인 Pointer가 적혀있어 읽습니다.
3. 이 포인터 주소로 이동해 실제 데이터 레코드를 가져오게 됩니다.

## 한 줄 요약
> 인덱스는 정렬된 Key 와 이에 대응하는 Pointer 매핑 테이블입니다.
> Key는 찾고자 하는 컬럼의 실제 값으로, 인덱스 구조 내에서 정렬된 상태로 존재하며 탐색 범위를 줄여줍니다.
> Pointer 는 해당 Key가 포함된 실제 행이 어디에 있는지 알려주는 주소값입니다.
> 쿼리 실행 시 DB가 인덱스 트리에서 Key를 통해 위치를 파악하고, Pointer을 따라 그 데이터 페이지에 접근하게 됩니다.


#### 추가 자료 및 첨언 ..

- MySQL(InnoDB)의 경우

기본은 **B+Tree** 이며 이 외에 **Full-text, R-Tree, Hash** 를 사용할 수 있습니다.

- PostgreSQL의 경우

기본은 **B-Tree** 이며, 이 외에 **Hash, GiST, GIN, BRIN, SP-GiST** 를 사용할 수 있습니다.

- Redis의 경우

기본은 **Hash Table/Skip List**이며, 이 외에 **List, Set, Sorted Set, STreams, String** 을 사용할 수 있습니다.

- H2 의 경우

**B- Tree**가 기본이며, 이 외에 인 메모리 모드로 **Hash**를 사용 가능합니다.
