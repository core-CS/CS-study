### Q. DB Connection Pool은 무엇이며 왜 필요할까?

DB 커넥션 풀은 애플리케이션과 DB 사이의 일부 커넥션을 미리 만들어두고, 필요에 따라 추가 생성하며 재사용하는 방식을 말합니다.

DB 커넥션은 생성 시 TCP 연결, 인증, 세션 생성 등의 비용이 발생하기 때문에 요청마다 새로 만들면 성능이 크게 저하될 수 있습니다.

커넥션 풀을 사용하면 이러한 비용을 줄이고, 재사용을 통해 응답 속도를 개선할 수 있습니다.

또한 동시에 생성 가능한 커넥션 수를 제한함으로써 DB에 과부하가 걸리는 것도 방지할 수 있습니다.

</br>

**DB Connection**

- DB를 사용하기 위해 DB와 애플리케이션 간 통신을 할 수 있는 수단
- DB Driver와 DB 연결 정보를 담은 url 필요
- java의 DB Connection은 JDBC를 사용, url 타입
    - JDBC : 자바 언어로 다양한 종류의 관계형 데이터베이스에 접속, SQL문 수행하고 처리하는 표준 SQL 인터페이스 API
    - JDBC 동작 과정 : 자바 웹 애플리케이션 → JDBC API → JDBC 드라이버 → DB

</br>

**DB Connection Pool**

- 웹 애플리케이션과 데이터베이스 사이의 연결을 미리 만들어두는 것 (필요에 따라 추가 + 재사용)
- 요청마다 DB 연결을 새로 생성하고 종료하는 과정은 TCP 연결, 인증 등의 비용이 발생하기 때문에 성능 저하의 원인이 됨
- → 이러한 오버헤드를 줄이고 응답 속도 개선 가능
- → 동시에 사용할 수 있는 커넥션 수를 제한하여 DB에 과도한 부하가 걸리는 것 방지 가능

</br>

**Connect 개수 조절**

부하 테스트를 진행했을 때, 다음과 같은 수치를 측정하면 개수를 조정할 수 있음
(보통은 DB 처리 능력보다 약간 적게 설정한다고 함)

- 응답 시간 느림 + 커넥션 대기 많음 : 커넥션 부족 상태이므로 커넥션 개수 증가
- DB CPU 100% 근접 : DB가 포화 상태이므로 커넥션 개수 감소
- TPS 증가 없음 : 커넥션 개수 유지 or 감소

</br>

**💡 Connection 획득 과정**

- 요청이 들어오면 애플리케이션은 커넥션 풀에서 커넥션을 가져옴
- 사용 가능한 커넥션이 있으면 즉시 반환
- 없을 경우:
    - 풀에 여유가 있으면 새로운 커넥션 생성
    - 최대 커넥션 수에 도달하면 **대기 상태**로 들어감

</br></br>

**💡 Timeout과 대기 처리**

- 커넥션이 모두 사용 중일 경우, 요청은 일정 시간 동안 대기
- 이때 timeout 설정이 없으면 쓰레드가 무한정 대기할 수 있음
    - 쓰레드 점유 지속
    - 요청 계속 쌓임
    - 결국 **쓰레드 풀 고갈** → 서비스 장애
- timeout 설정이 있으면
    - 일정 시간 이후 커넥션 획득 실패 (예외 발생)
    - 쓰레드 반환
    - 장애 확산 방지 가능

</br>

**Q. 그럼 쿼리 1개 = 커넥션 1개 인가요?**

아닙니다. 하나의 요청이나 트랜잭션 동안 하나의 커넥션을 사용하고, 작업이 끝나면 풀에 반납하는 방식입니다.

```
1. HTTP 요청
2. 커넥션 풀에서 커넥션 1개 가져옴
3. 해당 커넥션으로 여러 쿼리 실행
4. 트랜잭션 끝나면 커넥션 반납
```

만약 쿼리마다 커넥션을 가져온다면, 다음과 같은 문제가 생길 수 있음

- 풀에서 계속 빌림/반납 반복 → 오버헤드 증가
- 트랜잭션 유지 어려움
- DB 세션 일관성 깨짐
